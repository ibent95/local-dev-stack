# 002 · DuckDB & Apache Parquet Integration — Planning

**Date:** 2026-07-06
**Status:** Planned
**Type:** Planning

---

## Overview

Introduce a **DuckDB** analytical database service to the local-dev-stack,
with native support for **Apache Parquet** file queries. DuckDB is an
embedded, in-process analytical database optimized for OLAP workloads. Unlike
traditional databases, it can read Parquet files directly without importing
them, making it ideal for local data analysis workflows.

---

## 1 · Problem Statement

Currently, the LDS stack supports MySQL, Postgres, and MongoDB for
transactional workloads. For analytical queries over columnar data (Parquet,
CSV, JSON), developers must either:
- Install DuckDB locally (not reproducible)
- Use heavy tools like Apache Spark or Trino
- Write custom ETL pipelines

A managed DuckDB service within Docker Compose solves this by providing a
consistent, reproducible analytical query engine with Parquet support.

---

## 2 · Architecture

### 2.1 Why DuckDB Needs an API Wrapper

DuckDB is an **embedded, in-process database** — it has no native HTTP server,
no network protocol, and no client-server architecture. To expose it as a
service, we need a lightweight API wrapper.

**Options evaluated:**

| Approach | Pros | Cons |
|----------|------|------|
| **FastAPI (Python)** | Mature, async, DuckDB Python API is first-class | Python dependency, larger image |
| **Hono (Node/TS)** | Consistent with LDS APIs, lightweight | DuckDB Node API is less mature |
| **DuckDB CLI container** | No code needed | Interactive only, no HTTP API |
| **`ducktape`** | Ready-made HTTP wrapper | Third-party, may not be maintained |

**Recommended:** FastAPI (Python) — DuckDB's Python API is the most mature
and feature-complete. FastAPI provides async HTTP with minimal overhead.

> **Note:** This recommendation is pending confirmation. The user previously
> asked about implementing DuckDB "as a Service like MongoDB," which may
> indicate a preference for a different approach.

## 3 · Service Specification

### 3.1 Profile & Port

| Property | Value |
|----------|-------|
| Profile | `duckdb` |
| Default | Off (toggle: `LDS_ENABLE_DUCKDB=true`) |
| Included in `all` | Yes |
| API Port | `4450` |
| Container Name | `lds-duckdb` |

### 3.2 Docker Image

Build from a custom Dockerfile using Python base:

```dockerfile
FROM python:3.12-slim
RUN pip install duckdb fastapi uvicorn pydantic
WORKDIR /app
COPY api/ .
EXPOSE 4450
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "4450"]
```

### 3.3 Volume Mounts

| Host Path | Container Path | Purpose |
|-----------|---------------|---------|
| `data/duckdb/` | `/data` | Parquet/CSV files + DuckDB database |

### 3.4 API Endpoints

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/health` | Health check |
| `POST` | `/query` | Execute SQL query (returns JSON) |
| `GET` | `/tables` | List tables/views in the database |
| `GET` | `/parquet/info` | List Parquet files and their schemas |
| `POST` | `/parquet/query` | Query a Parquet file directly |

### 3.5 Example Requests

**Query the database:**
```json
POST /query
{
  "sql": "SELECT COUNT(*) as total FROM read_parquet('/data/sales.parquet')"
}
```

**List Parquet files:**
```json
GET /parquet/info
→ [
    { "file": "sales.parquet", "rows": 150000, "columns": ["date", "product", "revenue"] },
    { "file": "users.parquet", "rows": 5000, "columns": ["id", "name", "email"] }
  ]
```

**Query a Parquet file directly:**
```json
POST /parquet/query
{
  "file": "sales.parquet",
  "sql": "SELECT product, SUM(revenue) as total FROM '/data/sales.parquet' GROUP BY product ORDER BY total DESC LIMIT 10"
}
```

## 4 · DuckDB + Parquet Capabilities

### 4.1 Direct Parquet Reading

DuckDB can query Parquet files without importing them:

```sql
-- Query a single file
SELECT * FROM read_parquet('/data/file.parquet');

-- Query with filters (predicate pushdown)
SELECT * FROM read_parquet('/data/file.parquet') WHERE date > '2024-01-01';

-- Query multiple files (glob)
SELECT * FROM read_parquet('/data/*.parquet');

-- Auto-detect columns
DESCRIBE SELECT * FROM read_parquet('/data/file.parquet');
```

### 4.2 Parquet + SQL Analytics

```sql
-- Time series analysis
SELECT
  date_trunc('month', date) as month,
  SUM(revenue) as monthly_revenue,
  COUNT(*) as transactions
FROM read_parquet('/data/sales.parquet')
GROUP BY 1
ORDER BY 1;

-- Window functions over Parquet
SELECT *,
  RANK() OVER (PARTITION BY category ORDER BY revenue DESC) as rank
FROM read_parquet('/data/products.parquet');

-- Join Parquet with other data sources
SELECT o.*, c.name
FROM read_parquet('/data/orders.parquet') o
JOIN customers c ON o.customer_id = c.id;
```

### 4.3 Supported Formats

| Format | Read | Write | Notes |
|--------|------|-------|-------|
| Parquet | ✅ | ✅ | Columnar, compressed, ideal for analytics |
| CSV | ✅ | ✅ | Auto-detect delimiter, headers |
| JSON/JSONL | ✅ | ✅ | Nested structures supported |
| Arrow (IPC) | ✅ | ✅ | Zero-copy interchange |
| DuckDB (.duckdb) | ✅ | ✅ | Native format, ACID transactions |

## 5 · Docker Compose Entry

```yaml
# DuckDB — analytical database with Parquet support
duckdb:
  duckdb:
    profiles: [duckdb]
    build:
      context: configs/duckdb
    image: ${REGISTRY:-local}/local-dev-stack/duckdb:latest
    container_name: lds-duckdb
    restart: unless-stopped
    ports:
      - "${LDS_DUCKDB_PORT:-4450}:4450"
    volumes:
      - ./data/duckdb:/data
    environment:
      - TZ=${TZ:-Asia/Jakarta}
    healthcheck:
      test: ["CMD", "python", "-c", "import urllib.request; urllib.request.urlopen('http://localhost:4450/health')"]
      interval: 30s
      timeout: 5s
      retries: 3
      start_period: 10s
    mem_limit: 512m
    cpus: 1.0
```

## 10 · Open Questions

1. **Database persistence:** Should the DuckDB database file persist across
   container restarts? (Recommended: yes, via volume mount)

2. **Security:** Should the API require authentication for local dev?
   (Recommended: no, open access like other LDS services)

3. **Query limits:** Should we enforce query timeouts or result size limits?
   (Recommended: yes, prevent accidental OOM on large Parquet files)

4. **UI:** Should we add a web UI for DuckDB (query editor, result viewer)?
   (Deferred to future cycle)
